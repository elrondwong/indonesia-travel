更新日期：2026-06-25  
对应候选模板：[甘南莲宝叶则_订单候选模板.md](甘南莲宝叶则_订单候选模板.md)

这份文件解决一个更硬的问题：不是“候选表格式对不对”，而是“候选有没有覆盖完整旅程”。真实订单阶段，不能只填一两个看起来不错的酒店或门票；必须确认大交通、车辆、每天住宿和核心景区开放都被覆盖。

## 先看结论

普通检查：

```bash
python3 scripts/check_gannan_order_coverage.py
```

真实订单硬检查：

```bash
python3 scripts/check_gannan_order_coverage.py --strict
```

如果已经进入下单准备阶段，只认可 `shortlist`、`booked`、`backup`：

```bash
python3 scripts/check_gannan_order_coverage.py --strict --booking-ready
```

## 覆盖规则

| 类别 | 必须覆盖 | 为什么 |
|---|---|---|
| 大交通 | 去程和返程 | 防止只查到兰州、不查返程，或返程压缩 D8/D9 |
| 酒店 | D1-D8 每晚 | 9 天 8 晚路线不能漏夜；D0 可按航班另加 |
| 核心住宿区域 | 夏河、合作、扎尕那、唐克/若尔盖、阿坝县、合作/临夏 | 对应日出日落、莲宝叶则和返程分段 |
| 车辆 | 至少 1 个活跃方案，询价阶段建议 3 个 | 甘南/阿坝路线不能临时赌车 |
| 核心门票/开放 | 拉卜楞寺、扎尕那、花湖、黄河九曲、莲宝叶则 | 这些决定主路线和大片机位 |
| 扎尕那/莲宝叶则 | 建议至少 2 个来源 | 票价、开放范围、观光车和天气影响最强 |

## 状态口径

默认会把这些状态算作覆盖：

```text
candidate, shortlist, booked, watch, backup
```

`--booking-ready` 模式只认可：

```text
shortlist, booked, backup
```

所以前期收集候选时用普通模式；临近下单或最终核验时用 `--booking-ready`。

## 什么时候会失败

严格模式下会失败：

- 实际候选 CSV 不存在或为空。
- 大交通没有去程或返程。
- 酒店缺 D1-D8 任一晚。
- 车辆没有活跃方案。
- 门票/开放缺拉卜楞寺、扎尕那、花湖、黄河九曲、莲宝叶则任一项。

会给出警告但不一定失败：

- 大交通候选少于 2 个。
- 车辆候选少于 3 个。
- 扎尕那或阿坝县住宿候选少于 2 个。
- 扎尕那或莲宝叶则门票/开放来源少于 2 个。
- 车辆夜路政策没写清。

## 和一键总核验的关系

真实订单阶段：

```bash
python3 scripts/run_gannan_checks.py --actual
```

会依次检查：

- 出行信息 JSON 是否完整。
- 候选 CSV 字段是否完整。
- 候选覆盖率是否完整。
- 复核日志格式是否完整。
- 证据截图是否真实存在。
- 订单复核报告是否可生成。
